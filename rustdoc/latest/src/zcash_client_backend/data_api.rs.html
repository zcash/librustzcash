1//! # Utilities for Zcash wallet construction
2//!
3//! This module defines a set of APIs for wallet data persistence, and provides a suite of methods
4//! based upon these APIs that can be used to implement a fully functional Zcash wallet. At
5//! present, the interfaces provided here are built primarily around the use of a source of
6//! [`CompactBlock`] data such as the Zcash Light Client Protocol as defined in
7//! [ZIP 307](https://zips.z.cash/zip-0307) but they may be generalized to full-block use cases in
8//! the future.
9//!
10//! ## Important Concepts
11//!
12//! There are several important operations that a Zcash wallet must perform that distinguish Zcash
13//! wallet design from wallets for other cryptocurrencies.
14//!
15//! * Viewing Keys: Wallets based upon this module are built around the capabilities of Zcash
16//!   [`UnifiedFullViewingKey`]s; the wallet backend provides no facilities for the storage
17//!   of spending keys, and spending keys must be provided by the caller (or delegated to external
18//!   devices via the provided [`pczt`] functionality) in order to perform transaction creation
19//!   operations.
20//! * Blockchain Scanning: A Zcash wallet must download and trial-decrypt each transaction on the
21//!   Zcash blockchain using one or more Viewing Keys in order to find new shielded transaction
22//!   outputs (generally termed "notes") belonging to the wallet. The primary entrypoint for this
23//!   functionality is the [`scan_cached_blocks`] method. See the [`chain`] module for additional
24//!   details.
25//! * Witness Updates: In order to spend a shielded note, the wallet must be able to compute the
26//!   Merkle path to that note in the global note commitment tree. When [`scan_cached_blocks`] is
27//!   used to process a range of blocks, the note commitment tree is updated with the note
28//!   commitments for the blocks in that range.
29//! * Transaction Construction: The [`wallet`] module provides functions for creating Zcash
30//!   transactions that spend funds belonging to the wallet. Input selection for transaction
31//!   construction is performed automatically. When spending funds received at transparent
32//!   addresses, the caller is required to explicitly specify the set of addresses from which to
33//!   spend funds, in order to prevent inadvertent commingling of funds received at different
34//!   addresses; allowing such commingling would enable chain observers to identify those
35//!   addresses as belonging to the same user.
36//!
37//! ## Core Traits
38//!
39//! The utility functions described above depend upon four important traits defined in this
40//! module, which between them encompass the data storage requirements of a light wallet.
41//! The relevant traits are [`InputSource`], [`WalletRead`], [`WalletWrite`], and
42//! [`WalletCommitmentTrees`]. A complete implementation of the data storage layer for a wallet
43//! will include an implementation of all four of these traits. See the [`zcash_client_sqlite`]
44//! crate for a complete example of the implementation of these traits.
45//!
46//! ## Accounts
47//!
48//! The operation of the [`InputSource`], [`WalletRead`] and [`WalletWrite`] traits is built around
49//! the concept of a wallet having one or more accounts, with a unique `AccountId` for each
50//! account.
51//!
52//! An account identifier corresponds to at most a single [`UnifiedSpendingKey`]'s worth of spend
53//! authority, with the received and spent notes of that account tracked via the corresponding
54//! [`UnifiedFullViewingKey`]. Both received notes and change spendable by that spending authority
55//! (both the external and internal parts of that key, as defined by
56//! [ZIP 316](https://zips.z.cash/zip-0316)) will be interpreted as belonging to that account.
57//!
58//! [`CompactBlock`]: crate::proto::compact_formats::CompactBlock
59//! [`scan_cached_blocks`]: crate::data_api::chain::scan_cached_blocks
60//! [`zcash_client_sqlite`]: https://docs.rs/zcash_client_sqlite
61//! [`TransactionRequest`]: crate::zip321::TransactionRequest
62//! [`propose_shielding`]: crate::data_api::wallet::propose_shielding
63//! [`pczt`]: https://docs.rs/pczt

154/// An intent of representing spendable value to reach a certain targeted
155/// amount.
156///
157/// `AtLeast(Zatoshis)` refers to the amount of `Zatoshis` that can cover
158/// at minimum the given zatoshis that is conformed by the sum of spendable notes.
159///
160///
161/// Discussion: why not just use ``Zatoshis``?
162///
163/// the `Zatoshis` value isn't enough to explain intent when seeking to match a
164/// given a given amount. Is the value expressed in `Zatoshis` the ceiling value

169    AtLeast(Zatoshis),
170    AllFunds(MaxSpendMode),
171}

185/// Balance information for a value within a single pool in an account.
186#[derive(Debug, Clone, Copy, PartialEq, Eq)]
187pub struct Balance {
188    spendable_value: Zatoshis,
189    change_pending_confirmation: Zatoshis,
190    value_pending_spendability: Zatoshis,
191    uneconomic_value: Zatoshis,
192}
193
194impl Balance {
195    /// The [`Balance`] value having zero values for all its fields.
196    pub const ZERO: Self = Self {
197        spendable_value: Zatoshis::ZERO,
198        change_pending_confirmation: Zatoshis::ZERO,
199        value_pending_spendability: Zatoshis::ZERO,
200        uneconomic_value: Zatoshis::ZERO,
201    };
202
203    fn check_total_adding(&self, value: Zatoshis) -> Result<Zatoshis, BalanceError> {
204        (self.spendable_value
205            + self.change_pending_confirmation
206            + self.value_pending_spendability
207            + value)
208            .ok_or(BalanceError::Overflow)
209    }
210
211    /// Returns the value in the account that may currently be spent; it is possible to compute
212    /// witnesses for all the notes that comprise this value, and all of this value is confirmed to
213    /// the required confirmation depth.
214    pub fn spendable_value(&self) -> Zatoshis {
215        self.spendable_value
216    }
217
218    /// Adds the specified value to the spendable total, checking for overflow.
219    pub fn add_spendable_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
220        self.check_total_adding(value)?;
221        self.spendable_value = (self.spendable_value + value).unwrap();
222        Ok(())
223    }
224
225    /// Returns the value in the account of shielded change notes that do not yet have sufficient
226    /// confirmations to be spendable.
227    pub fn change_pending_confirmation(&self) -> Zatoshis {
228        self.change_pending_confirmation
229    }
230
231    /// Adds the specified value to the pending change total, checking for overflow.
232    pub fn add_pending_change_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
233        self.check_total_adding(value)?;
234        self.change_pending_confirmation = (self.change_pending_confirmation + value).unwrap();
235        Ok(())
236    }
237
238    /// Returns the value in the account of all remaining received notes that either do not have
239    /// sufficient confirmations to be spendable, or for which witnesses cannot yet be constructed
240    /// without additional scanning.
241    pub fn value_pending_spendability(&self) -> Zatoshis {
242        self.value_pending_spendability
243    }
244
245    /// Adds the specified value to the pending spendable total, checking for overflow.
246    pub fn add_pending_spendable_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
247        self.check_total_adding(value)?;
248        self.value_pending_spendability = (self.value_pending_spendability + value).unwrap();
249        Ok(())
250    }
251
252    /// Returns the value in the account of notes that have value less than the marginal
253    /// fee, and consequently cannot be spent except as a grace input.
254    pub fn uneconomic_value(&self) -> Zatoshis {
255        self.uneconomic_value
256    }
257
258    /// Adds the specified value to the uneconomic value total, checking for overflow.
259    pub fn add_uneconomic_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
260        self.uneconomic_value = (self.uneconomic_value + value).ok_or(BalanceError::Overflow)?;
261        Ok(())
262    }
263
264    /// Returns the total value of funds represented by this [`Balance`].
265    pub fn total(&self) -> Zatoshis {
266        (self.spendable_value + self.change_pending_confirmation + self.value_pending_spendability)
267            .expect("Balance cannot overflow MAX_MONEY")
268    }
269}
270
271impl core::ops::Add<Balance> for Balance {
272    type Output = Result<Balance, BalanceError>;
273
274    fn add(self, rhs: Balance) -> Self::Output {
275        let result = Balance {
276            spendable_value: (self.spendable_value + rhs.spendable_value)
277                .ok_or(BalanceError::Overflow)?,
278            change_pending_confirmation: (self.change_pending_confirmation
279                + rhs.change_pending_confirmation)
280                .ok_or(BalanceError::Overflow)?,
281            value_pending_spendability: (self.value_pending_spendability
282                + rhs.value_pending_spendability)
283                .ok_or(BalanceError::Overflow)?,
284            uneconomic_value: (self.uneconomic_value + rhs.uneconomic_value)
285                .ok_or(BalanceError::Overflow)?,
286        };
287
288        result.check_total_adding(Zatoshis::ZERO)?;
289
290        Ok(result)
291    }
292}

515/// A set of capabilities that a client account must provide.
516///
517/// An account represents a distinct set of viewing keys within the wallet; the keys for an account
518/// must not be shared with any other account in the wallet, and an application managing wallet
519/// accounts must ensure that it either maintains spending keys that can be used for spending _all_
520/// outputs detectable by the viewing keys of the account, or for none of them (i.e. the account is
521/// view-only.)
522///
523/// Balance information is available for any full-viewing-key based account; for an
524/// incoming-viewing-key only account balance cannot be determined because spends cannot be
525/// detected, and so balance-related APIs and APIs that rely upon spentness checks MUST be
526/// implemented to return errors if invoked for an IVK-only account.
527///
528/// For spending accounts in implementations that support the `transparent-key-import` feature,
529/// care must be taken to ensure that spending keys corresponding to every imported transparent
530/// address in an account are maintained by the application.
531pub trait Account {
532    type AccountId: Copy;
533
534    /// Returns the unique identifier for the account.
535    fn id(&self) -> Self::AccountId;
536
537    /// Returns the human-readable name for the account, if any has been configured.
538    fn name(&self) -> Option<&str>;
539
540    /// Returns whether this account is derived or imported, and the derivation parameters
541    /// if applicable.
542    fn source(&self) -> &AccountSource;
543
544    /// Returns whether the account is a spending account or a view-only account.
545    fn purpose(&self) -> AccountPurpose {
546        match self.source() {
547            AccountSource::Derived { derivation, .. } => AccountPurpose::Spending {
548                derivation: Some(derivation.clone()),
549            },
550            AccountSource::Imported { purpose, .. } => purpose.clone(),
551        }
552    }
553
554    /// Returns the UFVK that the wallet backend has stored for the account, if any.
555    ///
556    /// Accounts for which this returns `None` cannot be used in wallet contexts, because
557    /// they are unable to maintain an accurate balance.
558    fn ufvk(&self) -> Option<&UnifiedFullViewingKey>;
559
560    /// Returns the UIVK that the wallet backend has stored for the account.
561    ///
562    /// All accounts are required to have at least an incoming viewing key. This gives no
563    /// indication about whether an account can be used in a wallet context; for that, use
564    /// [`Account::ufvk`].
565    fn uivk(&self) -> UnifiedIncomingViewingKey;
566}

722/// A type representing the progress the wallet has made toward detecting all of the funds
723/// belonging to the wallet.
724///
725/// The window over which progress is computed spans from the wallet's birthday to the current
726/// chain tip. It is divided into two regions, the "Scan Window" which covers the region from the
727/// wallet recovery height to the current chain tip, and the "Recovery Window" which covers the

978            retention
979                .should_retain_orchard(&n)
980                .then(|| n.map_note(Note::Orchard))
981        }));
982
983        iter.collect()
984    }

1395/// A small query language for filtering notes belonging to an account.
1396///
1397/// A filter described using this language is applied to notes individually. It is primarily
1398/// intended for retrieval of account metadata in service of making determinations for how to
1399/// allocate change notes, and is not currently intended for use in broader note selection
1400/// contexts.
1401#[derive(Clone, Debug, PartialEq, Eq)]
1402pub enum NoteFilter {
1403    /// Selects notes having value greater than or equal to the provided value.
1404    ExceedsMinValue(Zatoshis),
1405    /// Selects notes having value greater than or equal to approximately the n'th percentile of
1406    /// previously sent notes in the account, irrespective of pool. The wrapped value must be in
1407    /// the range `1..=99`. The value `n` is respected in a best-effort fashion; results are likely
1408    /// to be inaccurate if the account has not yet completed scanning or if insufficient send data
1409    /// is available to establish a distribution.
1410    // TODO: it might be worthwhile to add an optional parameter here that can be used to ignore
1411    // low-valued (test/memo-only) sends when constructing the distribution to be drawn from.
1412    ExceedsPriorSendPercentile(BoundedU8<99>),
1413    /// Selects notes having value greater than or equal to the specified percentage of the account
1414    /// balance across all shielded pools. The wrapped value must be in the range `1..=99`
1415    ExceedsBalancePercentage(BoundedU8<99>),
1416    /// A note will be selected if it satisfies both of the specified conditions.
1417    ///
1418    /// If it is not possible to evaluate one of the conditions (for example,
1419    /// [`NoteFilter::ExceedsPriorSendPercentile`] cannot be evaluated if no sends have been
1420    /// performed) then that condition will be ignored. If neither condition can be evaluated,
1421    /// then the entire condition cannot be evaluated.
1422    Combine(Box<NoteFilter>, Box<NoteFilter>),
1423    /// A note will be selected if it satisfies the first condition; if it is not possible to
1424    /// evaluate that condition (for example, [`NoteFilter::ExceedsPriorSendPercentile`] cannot
1425    /// be evaluated if no sends have been performed) then the second condition will be used for
1426    /// evaluation.
1427    Attempt {
1428        condition: Box<NoteFilter>,
1429        fallback: Box<NoteFilter>,
1430    },
1431}
1432
1433impl NoteFilter {
1434    /// Constructs a [`NoteFilter::Combine`] query node.
1435    pub fn combine(l: NoteFilter, r: NoteFilter) -> Self {
1436        Self::Combine(Box::new(l), Box::new(r))
1437    }
1438
1439    /// Constructs a [`NoteFilter::Attempt`] query node.
1440    pub fn attempt(condition: NoteFilter, fallback: NoteFilter) -> Self {
1441        Self::Attempt {
1442            condition: Box::new(condition),
1443            fallback: Box::new(fallback),
1444        }
1445    }
1446}

2642/// This trait encapsulates the write capabilities required to update stored wallet data.
2643///
2644/// # Adding accounts
2645///
2646/// This trait provides several methods for adding accounts to the wallet data:
2647/// - [`WalletWrite::create_account`]
2648/// - [`WalletWrite::import_account_hd`]
2649/// - [`WalletWrite::import_account_ufvk`]
2650///
2651/// All of these methods take an [`AccountBirthday`]. The birthday height is defined as
2652/// the minimum block height that will be scanned for funds belonging to the wallet. If
2653/// `birthday.height()` is below the current chain tip, the account addition operation
2654/// will trigger a re-scan of the blocks at and above the provided height.
2655///
2656/// The order in which you call these methods will affect the resulting wallet structure:
2657/// - If only [`WalletWrite::create_account`] is used, the resulting accounts will have
2658///   sequential [ZIP 32] account indices within each given seed.
2659/// - If [`WalletWrite::import_account_hd`] is used to import accounts with non-sequential
2660///   ZIP 32 account indices from the same seed, a call to [`WalletWrite::create_account`]
2661///   will use the ZIP 32 account index just after the highest-numbered existing account.
2662/// - If an account is added to the wallet, and then a later call to one of the methods
2663///   would produce a UFVK that collides with that account on any FVK component (i.e.
2664///   Sapling, Orchard, or transparent), an error will be returned. This can occur in the
2665///   following cases:
2666///   - An account is created via [`WalletWrite::create_account`] with an auto-selected
2667///     ZIP 32 account index, and that index is later imported explicitly via either
2668///     [`WalletWrite::import_account_ufvk`] or [`WalletWrite::import_account_hd`].
2669///   - An account is imported via [`WalletWrite::import_account_ufvk`] or
2670///     [`WalletWrite::import_account_hd`], and then the ZIP 32 account index
2671///     corresponding to that account's UFVK is later imported either implicitly
2672///     via [`WalletWrite::create_account`], or explicitly via a call to
2673///     [`WalletWrite::import_account_ufvk`] or [`WalletWrite::import_account_hd`].
2674///
2675/// Note that an error will be returned on an FVK collision even if the UFVKs do not
2676/// match exactly, e.g. if they have different subsets of components.
2677///
2678/// An account is treated as having a single root of spending authority that spans the shielded and
2679/// transparent rules for the purpose of balance, transaction listing, and so forth. However,
2680/// transparent keys imported via [`WalletWrite::import_standalone_transparent_pubkey`] break this
2681/// abstraction slightly, so wallets using this API need to be cautious to enforce the invariant
2682/// that the wallet either maintains access to the keys required to spend **ALL** outputs received
2683/// by the account, or that it **DOES NOT** offer any spending capability for the account, i.e. the
2684/// account is treated as view-only for all user-facing operations.
2685///
2686/// A future change to this trait might introduce a method to "upgrade" an imported
2687/// account with derivation information. See [zcash/librustzcash#1284] for details.
2688///
2689/// Users of the `WalletWrite` trait should generally distinguish in their APIs and wallet UIs
2690/// between creating a new account, and importing an account that previously existed. By
2691/// convention, wallets should only allow a new account to be generated for a seed after confirmed
2692/// funds have been received by the newest existing account for that seed; this allows automated
2693/// account recovery to discover and recover all funds within a particular seed.
2694///
2695/// # Creating a new wallet
2696///
2697/// To create a new wallet:
2698/// - Generate a new [BIP 39] mnemonic phrase, using a crate like [`bip0039`].
2699/// - Derive the corresponding seed from the mnemonic phrase.
2700/// - Use [`WalletWrite::create_account`] with the resulting seed.
2701///
2702/// Callers should construct the [`AccountBirthday`] using [`AccountBirthday::from_treestate`] for
2703/// the block at height `chain_tip_height - 100`. Setting the birthday height to a tree state below
2704/// the pruning depth ensures that reorgs cannot cause funds intended for the wallet to be missed;
2705/// otherwise, if the chain tip height were used for the wallet birthday, a transaction targeted at
2706/// a height greater than the chain tip could be mined at a height below that tip as part of a
2707/// reorg.
2708///
2709/// # Restoring a wallet from backup
2710///
2711/// To restore a backed-up wallet:
2712/// - Derive the seed from its BIP 39 mnemonic phrase.
2713/// - Use [`WalletWrite::import_account_hd`] once for each ZIP 32 account index that the
2714///   user wants to restore.
2715/// - If the highest previously-used ZIP 32 account index was _not_ restored by the user,
2716///   remember this index separately as `index_max`. The first time the user wants to
2717///   generate a new account, use [`WalletWrite::import_account_hd`] to create the account
2718///   `index_max + 1`.
2719/// - [`WalletWrite::create_account`] can be used to generate subsequent new accounts in
2720///   the restored wallet.
2721///
2722/// Automated account recovery has not yet been implemented by this crate. A wallet app
2723/// that supports multiple accounts can implement it manually by tracking account balances
2724/// relative to [`WalletSummary::fully_scanned_height`], and creating new accounts as
2725/// funds appear in existing accounts.
2726///
2727/// If the number of accounts is known in advance, the wallet should create all accounts before
2728/// scanning the chain so that the scan can be done in a single pass for all accounts.
2729///
2730/// [ZIP 32]: https://zips.z.cash/zip-0032
2731/// [zcash/librustzcash#1284]: https://github.com/zcash/librustzcash/issues/1284
2732/// [BIP 39]: https://github.com/bitcoin/bips/blob/master/bip-0039.mediawiki

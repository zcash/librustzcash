1//! # Utilities for Zcash wallet construction
2//!
3//! This module defines a set of APIs for wallet data persistence, and provides a suite of methods
4//! based upon these APIs that can be used to implement a fully functional Zcash wallet. At
5//! present, the interfaces provided here are built primarily around the use of a source of
6//! [`CompactBlock`] data such as the Zcash Light Client Protocol as defined in
7//! [ZIP 307](https://zips.z.cash/zip-0307) but they may be generalized to full-block use cases in
8//! the future.
9//!
10//! ## Important Concepts
11//!
12//! There are several important operations that a Zcash wallet must perform that distinguish Zcash
13//! wallet design from wallets for other cryptocurrencies.
14//!
15//! * Viewing Keys: Wallets based upon this module are built around the capabilities of Zcash
16//!   [`UnifiedFullViewingKey`]s; the wallet backend provides no facilities for the storage
17//!   of spending keys, and spending keys must be provided by the caller in order to perform
18//!   transaction creation operations.
19//! * Blockchain Scanning: A Zcash wallet must download and trial-decrypt each transaction on the
20//!   Zcash blockchain using one or more Viewing Keys in order to find new shielded transaction
21//!   outputs (generally termed "notes") belonging to the wallet. The primary entrypoint for this
22//!   functionality is the [`scan_cached_blocks`] method. See the [`chain`] module for additional
23//!   details.
24//! * Witness Updates: In order to spend a shielded note, the wallet must be able to compute the
25//!   Merkle path to that note in the global note commitment tree. When [`scan_cached_blocks`] is
26//!   used to process a range of blocks, the note commitment tree is updated with the note
27//!   commitments for the blocks in that range.
28//! * Transaction Construction: The [`wallet`] module provides functions for creating Zcash
29//!   transactions that spend funds belonging to the wallet.
30//!
31//! ## Core Traits
32//!
33//! The utility functions described above depend upon four important traits defined in this
34//! module, which between them encompass the data storage requirements of a light wallet.
35//! The relevant traits are [`InputSource`], [`WalletRead`], [`WalletWrite`], and
36//! [`WalletCommitmentTrees`]. A complete implementation of the data storage layer for a wallet
37//! will include an implementation of all four of these traits. See the [`zcash_client_sqlite`]
38//! crate for a complete example of the implementation of these traits.
39//!
40//! ## Accounts
41//!
42//! The operation of the [`InputSource`], [`WalletRead`] and [`WalletWrite`] traits is built around
43//! the concept of a wallet having one or more accounts, with a unique `AccountId` for each
44//! account.
45//!
46//! An account identifier corresponds to at most a single [`UnifiedSpendingKey`]'s worth of spend
47//! authority, with the received and spent notes of that account tracked via the corresponding
48//! [`UnifiedFullViewingKey`]. Both received notes and change spendable by that spending authority
49//! (both the external and internal parts of that key, as defined by
50//! [ZIP 316](https://zips.z.cash/zip-0316)) will be interpreted as belonging to that account.
51//!
52//! [`CompactBlock`]: crate::proto::compact_formats::CompactBlock
53//! [`scan_cached_blocks`]: crate::data_api::chain::scan_cached_blocks
54//! [`zcash_client_sqlite`]: https://crates.io/crates/zcash_client_sqlite
55//! [`TransactionRequest`]: crate::zip321::TransactionRequest
56//! [`propose_shielding`]: crate::data_api::wallet::propose_shielding

157/// Balance information for a value within a single pool in an account.
158#[derive(Debug, Clone, Copy, PartialEq, Eq)]
159pub struct Balance {
160    spendable_value: Zatoshis,
161    change_pending_confirmation: Zatoshis,
162    value_pending_spendability: Zatoshis,
163}
164
165impl Balance {
166    /// The [`Balance`] value having zero values for all its fields.
167    pub const ZERO: Self = Self {
168        spendable_value: Zatoshis::ZERO,
169        change_pending_confirmation: Zatoshis::ZERO,
170        value_pending_spendability: Zatoshis::ZERO,
171    };
172
173    fn check_total_adding(&self, value: Zatoshis) -> Result<Zatoshis, BalanceError> {
174        (self.spendable_value
175            + self.change_pending_confirmation
176            + self.value_pending_spendability
177            + value)
178            .ok_or(BalanceError::Overflow)
179    }
180
181    /// Returns the value in the account that may currently be spent; it is possible to compute
182    /// witnesses for all the notes that comprise this value, and all of this value is confirmed to
183    /// the required confirmation depth.
184    pub fn spendable_value(&self) -> Zatoshis {
185        self.spendable_value
186    }
187
188    /// Adds the specified value to the spendable total, checking for overflow.
189    pub fn add_spendable_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
190        self.check_total_adding(value)?;
191        self.spendable_value = (self.spendable_value + value).unwrap();
192        Ok(())
193    }
194
195    /// Returns the value in the account of shielded change notes that do not yet have sufficient
196    /// confirmations to be spendable.
197    pub fn change_pending_confirmation(&self) -> Zatoshis {
198        self.change_pending_confirmation
199    }
200
201    /// Adds the specified value to the pending change total, checking for overflow.
202    pub fn add_pending_change_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
203        self.check_total_adding(value)?;
204        self.change_pending_confirmation = (self.change_pending_confirmation + value).unwrap();
205        Ok(())
206    }
207
208    /// Returns the value in the account of all remaining received notes that either do not have
209    /// sufficient confirmations to be spendable, or for which witnesses cannot yet be constructed
210    /// without additional scanning.
211    pub fn value_pending_spendability(&self) -> Zatoshis {
212        self.value_pending_spendability
213    }
214
215    /// Adds the specified value to the pending spendable total, checking for overflow.
216    pub fn add_pending_spendable_value(&mut self, value: Zatoshis) -> Result<(), BalanceError> {
217        self.check_total_adding(value)?;
218        self.value_pending_spendability = (self.value_pending_spendability + value).unwrap();
219        Ok(())
220    }
221
222    /// Returns the total value of funds represented by this [`Balance`].
223    pub fn total(&self) -> Zatoshis {
224        (self.spendable_value + self.change_pending_confirmation + self.value_pending_spendability)
225            .expect("Balance cannot overflow MAX_MONEY")
226    }
227}

387/// The kinds of accounts supported by `zcash_client_backend`.
388#[derive(Clone, Debug, PartialEq, Eq, Hash)]
389pub enum AccountSource {
390    /// An account derived from a known seed.
391    Derived {
392        derivation: Zip32Derivation,
393        key_source: Option<String>,
394    },
395
396    /// An account imported from a viewing key.
397    Imported {
398        purpose: AccountPurpose,
399        key_source: Option<String>,
400    },
401}
402
403impl AccountSource {
404    /// Returns the key derivation metadata for the account source, if any is available.
405    pub fn key_derivation(&self) -> Option<&Zip32Derivation> {
406        match self {
407            AccountSource::Derived { derivation, .. } => Some(derivation),
408            AccountSource::Imported {
409                purpose: AccountPurpose::Spending { derivation },
410                ..
411            } => derivation.as_ref(),
412            _ => None,
413        }
414    }
415
416    /// Returns the application-level key source identifier.
417    pub fn key_source(&self) -> Option<&str> {
418        match self {
419            AccountSource::Derived { key_source, .. } => key_source.as_ref().map(|s| s.as_str()),
420            AccountSource::Imported { key_source, .. } => key_source.as_ref().map(|s| s.as_str()),
421        }
422    }
423}

425/// A set of capabilities that a client account must provide.
426pub trait Account {
427    type AccountId: Copy;
428
429    /// Returns the unique identifier for the account.
430    fn id(&self) -> Self::AccountId;
431
432    /// Returns the human-readable name for the account, if any has been configured.
433    fn name(&self) -> Option<&str>;
434
435    /// Returns whether this account is derived or imported, and the derivation parameters
436    /// if applicable.
437    fn source(&self) -> &AccountSource;
438
439    /// Returns whether the account is a spending account or a view-only account.
440    fn purpose(&self) -> AccountPurpose {
441        match self.source() {
442            AccountSource::Derived { derivation, .. } => AccountPurpose::Spending {
443                derivation: Some(derivation.clone()),
444            },
445            AccountSource::Imported { purpose, .. } => purpose.clone(),
446        }
447    }
448
449    /// Returns the UFVK that the wallet backend has stored for the account, if any.
450    ///
451    /// Accounts for which this returns `None` cannot be used in wallet contexts, because
452    /// they are unable to maintain an accurate balance.
453    fn ufvk(&self) -> Option<&UnifiedFullViewingKey>;
454
455    /// Returns the UIVK that the wallet backend has stored for the account.
456    ///
457    /// All accounts are required to have at least an incoming viewing key. This gives no
458    /// indication about whether an account can be used in a wallet context; for that, use
459    /// [`Account::ufvk`].
460    fn uivk(&self) -> UnifiedIncomingViewingKey;
461}

593/// A type representing the progress the wallet has made toward detecting all of the funds
594/// belonging to the wallet.
595///
596/// The window over which progress is computed spans from the wallet's birthday to the current
597/// chain tip. It is divided into two regions, the "Scan Window" which covers the region from the
598/// wallet recovery height to the current chain tip, and the "Recovery Window" which covers the
599/// range from the wallet birthday to the wallet recovery height. If no wallet recovery height is
600/// available, the scan window will cover the entire range from the wallet birthday to the chain
601/// tip.
602///
603/// Progress for both scanning and recovery is represented in terms of the ratio between notes
604/// scanned and the total number of notes added to the chain in the relevant window. This ratio
605/// should only be used to compute progress percentages for display, and the numerator and
606/// denominator should not be treated as authoritative note counts. In the case that there are no
607/// notes in a given block range, the denominator of these values will be zero, so callers should always
608/// use checked division when converting the resulting values to percentages.
609#[derive(Debug, Clone, Copy, PartialEq, Eq)]
610pub struct Progress {
611    scan: Ratio<u64>,
612    recovery: Option<Ratio<u64>>,
613}
614
615impl Progress {
616    /// Constructs a new progress value from its constituent parts.
617    pub fn new(scan: Ratio<u64>, recovery: Option<Ratio<u64>>) -> Self {
618        Self { scan, recovery }
619    }
620
621    /// Returns the progress the wallet has made in scanning blocks for shielded notes belonging to
622    /// the wallet between the wallet recovery height (or the wallet birthday if no recovery height
623    /// is set) and the chain tip.
624    pub fn scan(&self) -> Ratio<u64> {
625        self.scan
626    }
627
628    /// Returns the progress the wallet has made in scanning blocks for shielded notes belonging to
629    /// the wallet between the wallet birthday and the block height at which recovery from seed was
630    /// initiated.
631    ///
632    /// Returns `None` if no recovery height is set for the wallet.
633    pub fn recovery(&self) -> Option<Ratio<u64>> {
634        self.recovery
635    }
636}

1128/// A small query language for filtering notes belonging to an account.
1129///
1130/// A filter described using this language is applied to notes individually. It is primarily
1131/// intended for retrieval of account metadata in service of making determinations for how to
1132/// allocate change notes, and is not currently intended for use in broader note selection
1133/// contexts.

1135pub enum NoteFilter {
1136    /// Selects notes having value greater than or equal to the provided value.
1137    ExceedsMinValue(Zatoshis),
1138    /// Selects notes having value greater than or equal to approximately the n'th percentile of
1139    /// previously sent notes in the account, irrespective of pool. The wrapped value must be in
1140    /// the range `1..=99`. The value `n` is respected in a best-effort fashion; results are likely
1141    /// to be inaccurate if the account has not yet completed scanning or if insufficient send data
1142    /// is available to establish a distribution.

1144    // low-valued (test/memo-only) sends when constructing the distribution to be drawn from.
1145    ExceedsPriorSendPercentile(BoundedU8<99>),
1146    /// Selects notes having value greater than or equal to the specified percentage of the account
1147    /// balance across all shielded pools. The wrapped value must be in the range `1..=99`
1148    ExceedsBalancePercentage(BoundedU8<99>),
1149    /// A note will be selected if it satisfies both of the specified conditions.
1150    ///
1151    /// If it is not possible to evaluate one of the conditions (for example,
1152    /// [`NoteFilter::ExceedsPriorSendPercentile`] cannot be evaluated if no sends have been
1153    /// performed) then that condition will be ignored. If neither condition can be evaluated,
1154    /// then the entire condition cannot be evaluated.

1156    /// A note will be selected if it satisfies the first condition; if it is not possible to
1157    /// evaluate that condition (for example, [`NoteFilter::ExceedsPriorSendPercentile`] cannot
1158    /// be evaluated if no sends have been performed) then the second condition will be used for

1163    },

2279/// This trait encapsulates the write capabilities required to update stored wallet data.
2280///
2281/// # Adding accounts
2282///
2283/// This trait provides several methods for adding accounts to the wallet data:
2284/// - [`WalletWrite::create_account`]
2285/// - [`WalletWrite::import_account_hd`]
2286/// - [`WalletWrite::import_account_ufvk`]
2287///
2288/// All of these methods take an [`AccountBirthday`]. The birthday height is defined as
2289/// the minimum block height that will be scanned for funds belonging to the wallet. If
2290/// `birthday.height()` is below the current chain tip, the account addition operation
2291/// will trigger a re-scan of the blocks at and above the provided height.
2292///
2293/// The order in which you call these methods will affect the resulting wallet structure:
2294/// - If only [`WalletWrite::create_account`] is used, the resulting accounts will have
2295///   sequential [ZIP 32] account indices within each given seed.
2296/// - If [`WalletWrite::import_account_hd`] is used to import accounts with non-sequential
2297///   ZIP 32 account indices from the same seed, a call to [`WalletWrite::create_account`]
2298///   will use the ZIP 32 account index just after the highest-numbered existing account.
2299/// - If an account is added to the wallet, and then a later call to one of the methods
2300///   would produce a UFVK that collides with that account on any FVK component (i.e.
2301///   Sapling, Orchard, or transparent), an error will be returned. This can occur in the
2302///   following cases:
2303///   - An account is created via [`WalletWrite::create_account`] with an auto-selected
2304///     ZIP 32 account index, and that index is later imported explicitly via either
2305///     [`WalletWrite::import_account_ufvk`] or [`WalletWrite::import_account_hd`].
2306///   - An account is imported via [`WalletWrite::import_account_ufvk`] or
2307///     [`WalletWrite::import_account_hd`], and then the ZIP 32 account index
2308///     corresponding to that account's UFVK is later imported either implicitly
2309///     via [`WalletWrite::create_account`], or explicitly via a call to
2310///     [`WalletWrite::import_account_ufvk`] or [`WalletWrite::import_account_hd`].
2311///
2312/// Note that an error will be returned on an FVK collision even if the UFVKs do not
2313/// match exactly, e.g. if they have different subsets of components.
2314///
2315/// A future change to this trait might introduce a method to "upgrade" an imported
2316/// account with derivation information. See [zcash/librustzcash#1284] for details.
2317///
2318/// Users of the `WalletWrite` trait should generally distinguish in their APIs and wallet
2319/// UIs between creating a new account, and importing an account that previously existed.
2320/// By convention, wallets should only allow a new account to be generated after confirmed
2321/// funds have been received by the newest existing account; this allows automated account
2322/// recovery to discover and recover all funds within a particular seed.
2323///
2324/// # Creating a new wallet
2325///
2326/// To create a new wallet:
2327/// - Generate a new [BIP 39] mnemonic phrase, using a crate like [`bip0039`].
2328/// - Derive the corresponding seed from the mnemonic phrase.
2329/// - Use [`WalletWrite::create_account`] with the resulting seed.
2330///
2331/// Callers should construct the [`AccountBirthday`] using [`AccountBirthday::from_treestate`] for
2332/// the block at height `chain_tip_height - 100`. Setting the birthday height to a tree state below
2333/// the pruning depth ensures that reorgs cannot cause funds intended for the wallet to be missed;
2334/// otherwise, if the chain tip height were used for the wallet birthday, a transaction targeted at
2335/// a height greater than the chain tip could be mined at a height below that tip as part of a
2336/// reorg.
2337///
2338/// # Restoring a wallet from backup
2339///
2340/// To restore a backed-up wallet:
2341/// - Derive the seed from its BIP 39 mnemonic phrase.
2342/// - Use [`WalletWrite::import_account_hd`] once for each ZIP 32 account index that the
2343///   user wants to restore.
2344/// - If the highest previously-used ZIP 32 account index was _not_ restored by the user,
2345///   remember this index separately as `index_max`. The first time the user wants to
2346///   generate a new account, use [`WalletWrite::import_account_hd`] to create the account
2347///   `index_max + 1`.
2348/// - [`WalletWrite::create_account`] can be used to generate subsequent new accounts in
2349///   the restored wallet.
2350///
2351/// Automated account recovery has not yet been implemented by this crate. A wallet app
2352/// that supports multiple accounts can implement it manually by tracking account balances
2353/// relative to [`WalletSummary::fully_scanned_height`], and creating new accounts as
2354/// funds appear in existing accounts.
2355///
2356/// If the number of accounts is known in advance, the wallet should create all accounts before
2357/// scanning the chain so that the scan can be done in a single pass for all accounts.
2358///
2359/// [ZIP 32]: https://zips.z.cash/zip-0032
2360/// [zcash/librustzcash#1284]: https://github.com/zcash/librustzcash/issues/1284
2361/// [BIP 39]: https://github.com/bitcoin/bips/blob/master/bip-0039.mediawiki
2362/// [`bip0039`]: https://crates.io/crates/bip0039
2363#[cfg_attr(feature = "test-dependencies", delegatable_trait)]
2364pub trait WalletWrite: WalletRead {
2365    /// The type of identifiers used to look up transparent UTXOs.
2366    type UtxoRef;

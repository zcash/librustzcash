1//! Documentation about the wallet database structure.
2//!
3//! The database structure is managed by [`crate::wallet::init::init_wallet_db`], which
4//! applies migrations (defined in `crate::wallet::init::migrations`) that produce the
5//! current structure.
6//!
7//! The SQL code in this module's constants encodes the current database structure, as
8//! represented internally by SQLite. We do not use these constants at runtime; instead we
9//! check the output of the migrations in `crate::wallet::init::tests::verify_schema`, to
10//! pin the expected database structure.

86/// Stores ephemeral transparent addresses used for ZIP 320.
87///
88/// For each account, these addresses are allocated sequentially by address index under scope 2
89/// (`TransparentKeyScope::EPHEMERAL`) at the "change" level of the BIP 32 address hierarchy.
90/// The ephemeral addresses stored in the table are exactly the "reserved" ephemeral addresses
91/// (that is addresses that have been allocated for use in a ZIP 320 transaction proposal), plus
92/// the addresses at the next [`GAP_LIMIT`] indices.
93///
94/// Addresses are never removed. New ones should only be reserved via the
95/// `WalletWrite::reserve_next_n_ephemeral_addresses` API. All of the addresses in the table
96/// should be scanned for incoming funds.
97///
98/// ### Columns
99/// - `address` contains the string (Base58Check) encoding of a transparent P2PKH address.
100/// - `used_in_tx` indicates that the address has been used by this wallet in a transaction (which
101///   has not necessarily been mined yet). This should only be set once, when the txid is known.
102/// - `seen_in_tx` is non-null iff an output to the address has been seed in a transaction observed
103///   on the network and passed to `store_decrypted_tx`. The transaction may have been sent by this
104//    wallet or another one using the same seed, or by a TEX address recipient sending back the
105///   funds. This is used to advance the "gap", as well as to heuristically reduce the chance of
106///   address reuse collisions with another wallet using the same seed.
107///
108/// It is an external invariant that within each account:
109/// - the address indices are contiguous and start from 0;
110/// - the last [`GAP_LIMIT`] addresses have `used_in_tx` and `seen_in_tx` both NULL.
111///
112/// All but the last [`GAP_LIMIT`] addresses are defined to be "reserved" addresses. Since the next
113/// index to reserve is determined by dead reckoning from the last stored address, we use dummy
114/// entries having `NULL` for the value of the `address` column after the maximum valid index in
115/// order to allow the last [`GAP_LIMIT`] addresses at the end of the index range to be used.
116///
117/// Note that the fact that `used_in_tx` references a specific transaction is just a debugging aid.
118/// The same is mostly true of `seen_in_tx`, but we also take into account whether the referenced
119/// transaction is unmined in order to determine the last index that is safe to reserve.
120pub(super) const TABLE_EPHEMERAL_ADDRESSES: &str = r#"
121CREATE TABLE ephemeral_addresses (
122    account_id INTEGER NOT NULL,
123    address_index INTEGER NOT NULL,
124    -- nullability of this column is controlled by the index_range_and_address_nullity check
125    address TEXT,
126    used_in_tx INTEGER,
127    seen_in_tx INTEGER,
128    FOREIGN KEY (account_id) REFERENCES accounts(id),
129    FOREIGN KEY (used_in_tx) REFERENCES transactions(id_tx),
130    FOREIGN KEY (seen_in_tx) REFERENCES transactions(id_tx),
131    PRIMARY KEY (account_id, address_index),
132    CONSTRAINT ephemeral_addr_uniq UNIQUE (address),
133    CONSTRAINT used_implies_seen CHECK (
134        used_in_tx IS NULL OR seen_in_tx IS NOT NULL
135    ),
136    CONSTRAINT index_range_and_address_nullity CHECK (
137        (address_index BETWEEN 0 AND 0x7FFFFFFF AND address IS NOT NULL) OR
138        (address_index BETWEEN 0x80000000 AND 0x7FFFFFFF + 20 AND address IS NULL AND used_in_tx IS NULL AND seen_in_tx IS NULL)
139    )
140) WITHOUT ROWID"#;
141// Hexadecimal integer literals were added in SQLite version 3.8.6 (2014-08-15).
142// libsqlite3-sys requires at least version 3.14.0.
143// "WITHOUT ROWID" tells SQLite to use a clustered index on the (composite) primary key.
144const_assert_eq!(GAP_LIMIT, 20);

162/// Stores the wallet's transactions.
163///
164/// Any transactions that the wallet observes as "belonging to" one of the accounts in
165/// [`TABLE_ACCOUNTS`] may be tracked in this table. As a result, this table may contain
166/// data that is not recoverable from the chain (for example, transactions created by the
167/// wallet that expired before being mined).
168///
169/// ### Columns
170/// - `created`: The time at which the transaction was created as a string in the format
171///   `yyyy-MM-dd HH:mm:ss.fffffffzzz`.
172/// - `block`: stores the height (in the wallet's chain view) of the mined block containing the
173///   transaction. It is `NULL` for transactions that have not yet been observed in scanned blocks,
174///   including transactions in the mempool or that have expired.
175/// - `mined_height`: stores the height (in the wallet's chain view) of the mined block containing
176///   the transaction. It is present to allow the block height for a retrieved transaction to be
177///   stored without requiring that the entire block containing the transaction be scanned; the
178///   foreign key constraint on `block` prevents that column from being populated prior to complete
179///   scanning of the block. This is constrained to be equal to the `block` column if `block` is
180///   non-null.
181/// - `target_height`: stores the target height for which the transaction was constructed, if
182///   known. This will ordinarily be null for transactions discovered via chain scanning; it
183///   will only be set for transactions created using this wallet specifically, and not any
184///   other wallet that uses the same seed (including previous installations of the same
185///   wallet application.)
186pub(super) const TABLE_TRANSACTIONS: &str = r#"
187CREATE TABLE "transactions" (
188    id_tx INTEGER PRIMARY KEY,
189    txid BLOB NOT NULL UNIQUE,
190    created TEXT,
191    block INTEGER,
192    mined_height INTEGER,
193    tx_index INTEGER,
194    expiry_height INTEGER,
195    raw BLOB,
196    fee INTEGER,
197    target_height INTEGER,
198    FOREIGN KEY (block) REFERENCES blocks(height),
199    CONSTRAINT height_consistency CHECK (block IS NULL OR mined_height = block)
200)"#;

232/// A junction table between received Sapling notes and the transactions that spend them.
233///
234/// Only one mined transaction can spend a note. However, transactions created by the
235/// wallet may expire before being mined, and the wallet still tracks the fact that the
236/// user created the transaction. The junction table enables the "spent-in" relationship
237/// between notes and expired transactions to be preserved; note spent-ness is determined
238/// by joining this table with [`TABLE_TRANSACTIONS`] and then filtering out transactions
239/// where either `transactions.block` is non-null, or `transactions.expiry_height` is not
240/// greater than the wallet's view of the chain tip.
241pub(super) const TABLE_SAPLING_RECEIVED_NOTE_SPENDS: &str = "
242CREATE TABLE sapling_received_note_spends (
243    sapling_received_note_id INTEGER NOT NULL,
244    transaction_id INTEGER NOT NULL,
245    FOREIGN KEY (sapling_received_note_id)
246        REFERENCES sapling_received_notes(id)
247        ON DELETE CASCADE,
248    FOREIGN KEY (transaction_id)
249        -- We do not delete transactions, so this does not cascade
250        REFERENCES transactions(id_tx),
251    UNIQUE (sapling_received_note_id, transaction_id)
252)";

302/// Stores the transparent outputs received by the wallet.
303///
304/// Originally this table only stored the current UTXO set (as of latest refresh), and the
305/// table was cleared prior to loading in the latest UTXO set. We now upsert instead of
306/// insert into the database, meaning that spent outputs are left in the database. This
307/// makes it similar to the `*_received_notes` tables in that it can store history.
308/// Depending upon how transparent TXOs for the wallet are discovered, the following
309/// may be true:
310/// - The table may have incomplete contents for recovered-from-seed wallets.
311/// - The table may have inconsistent contents for seeds loaded into multiple wallets
312///   simultaneously.
313/// - The wallet's transparent balance may be incorrect prior to "transaction enhancement"
314///   (downloading the full transaction containing the transparent output spend).
315///
316/// ### Columns:
317/// - `id`: Primary key
318/// - `transaction_id`: Reference to the transaction in which this TXO was created
319/// - `output_index`: The output index of this TXO in the transaction referred to by `transaction_id`
320/// - `account_id`: The account that controls spend authority for this TXO
321/// - `address`: The address to which this TXO was sent. We store this address to make querying
322///   for UTXOs for a single address easier, because when shielding we always select UTXOs
323///   for only a single address at a time to prevent linking addresses in the shielding
324///   transaction.
325/// - `script`: The full txout script
326/// - `value_zat`: The value of the TXO in zatoshis
327/// - `max_observed_unspent_height`: The maximum block height at which this TXO was either
328///   observed to be a member of the UTXO set at the start of the block, or observed
329///   to be an output of a transaction mined in the block. This is intended to be used to
330///   determine when the TXO is no longer a part of the UTXO set, in the case that the
331///   transaction that spends it is not detected by the wallet.
332pub(super) const TABLE_TRANSPARENT_RECEIVED_OUTPUTS: &str = r#"
333CREATE TABLE transparent_received_outputs (
334    id INTEGER PRIMARY KEY,
335    transaction_id INTEGER NOT NULL,
336    output_index INTEGER NOT NULL,
337    account_id INTEGER NOT NULL,
338    address TEXT NOT NULL,
339    script BLOB NOT NULL,
340    value_zat INTEGER NOT NULL,
341    max_observed_unspent_height INTEGER,
342    FOREIGN KEY (transaction_id) REFERENCES transactions(id_tx),
343    FOREIGN KEY (account_id) REFERENCES accounts(id),
344    CONSTRAINT transparent_output_unique UNIQUE (transaction_id, output_index)
345)"#;
346pub(super) const INDEX_TRANSPARENT_RECEIVED_OUTPUTS_ACCOUNT_ID: &str = r#"
347CREATE INDEX idx_transparent_received_outputs_account_id
348ON "transparent_received_outputs" (account_id)"#;

394/// Stores the outputs of transactions created by the wallet.
395///
396/// Unlike with outputs received by the wallet, we store sent outputs for all pools in
397/// this table, distinguished by the `output_pool` column. The information we want to
398/// record for sent outputs is the same across all pools, whereas for received outputs we
399/// want to cache pool-specific data.
400///
401/// ### Columns
402/// - `(tx, output_pool, output_index)` collectively identify a transaction output.
403/// - `from_account_id`: the ID of the account that created the transaction.
404///   - On recover-from-seed or when scanning by UFVK, this will be either the account
405///     that decrypted the output, or one of the accounts that funded the transaction.
406/// - `to_address`: the address of the external recipient of this output, or `NULL` if the
407///   output was received by the wallet.
408/// - `to_account_id`: the ID of the account that received this output, or `NULL` if the
409///   output was for an external recipient.
410/// - `value`: the value of the output in zatoshis.
411/// - `memo`: the memo bytes associated with this output, if known.
412///   - This is always `NULL` for transparent outputs.
413///   - This will be set for all shielded outputs of transactions created by the wallet.
414///   - On recover-from-seed or when scanning by UFVK, this will only be set for shielded
415///     outputs after post-scanning transaction enhancement. For shielded notes sent to
416///     external recipients, the transaction needs to have been created with an
417///     [`OvkPolicy`] using a known OVK.
418///
419/// [`OvkPolicy`]: zcash_client_backend::wallet::OvkPolicy
420pub(super) const TABLE_SENT_NOTES: &str = r#"
421CREATE TABLE "sent_notes" (
422    id INTEGER PRIMARY KEY,
423    tx INTEGER NOT NULL,
424    output_pool INTEGER NOT NULL,
425    output_index INTEGER NOT NULL,
426    from_account_id INTEGER NOT NULL,
427    to_address TEXT,
428    to_account_id INTEGER,
429    value INTEGER NOT NULL,
430    memo BLOB,
431    FOREIGN KEY (tx) REFERENCES transactions(id_tx),
432    FOREIGN KEY (from_account_id) REFERENCES accounts(id),
433    FOREIGN KEY (to_account_id) REFERENCES accounts(id),
434    CONSTRAINT tx_output UNIQUE (tx, output_pool, output_index),
435    CONSTRAINT note_recipient CHECK (
436        (to_address IS NOT NULL) OR (to_account_id IS NOT NULL)
437    )
438)"#;

634/// A map from nullifiers to the transaction they were observed in.
635///
636/// The purpose of this map is to allow non-linear scanning. If the wallet scans a block
637/// range `Y..Z` that leaves a gap between the wallet's birthday height and `Y`, then the
638/// wallet must assume that any nullifier observed in `Y..Z` might be spending one of its
639/// notes (that it has not yet observed), otherwise it will fail to detect those spends
640/// and report a too-large balance. Once the wallet has scanned every block between its
641/// birthday height and `Y`, the nullifier map contents up to `Z` is no longer necessary
642/// and can be dropped.
643///
644/// The map stores transaction locators instead of transaction IDs for efficiency. SQLite
645/// will represent the transaction locator in at most 6 bytes, so a transaction that only
646/// spends one shielded note will incur a 12-byte overhead (across both this table and
647/// [`TABLE_TX_LOCATOR_MAP`]), but each additional spent note in a transaction saves 26
648/// bytes.
649pub(super) const TABLE_NULLIFIER_MAP: &str = "
650CREATE TABLE nullifier_map (
651    spend_pool INTEGER NOT NULL,
652    nf BLOB NOT NULL,
653    block_height INTEGER NOT NULL,
654    tx_index INTEGER NOT NULL,
655    CONSTRAINT tx_locator
656        FOREIGN KEY (block_height, tx_index)
657        REFERENCES tx_locator_map(block_height, tx_index)
658        ON DELETE CASCADE
659        ON UPDATE RESTRICT,
660    CONSTRAINT nf_uniq UNIQUE (spend_pool, nf)
661)";
662pub(super) const INDEX_NF_MAP_LOCATOR_IDX: &str =
663    r#"CREATE INDEX nf_map_locator_idx ON nullifier_map(block_height, tx_index)"#;

749pub(super) const VIEW_TRANSACTIONS: &str = "
750CREATE VIEW v_transactions AS
751WITH
752notes AS (
753    -- Outputs received in this transaction
754    SELECT ro.account_id              AS account_id,
755           transactions.mined_height  AS mined_height,
756           transactions.txid          AS txid,
757           ro.pool                    AS pool,
758           id_within_pool_table,
759           ro.value                   AS value,
760           ro.value                   AS received_value,
761           0                          AS spent_value,
762           0                          AS spent_note_count,
763           CASE
764                WHEN ro.is_change THEN 1
765                ELSE 0
766           END AS change_note_count,
767           CASE
768                WHEN ro.is_change THEN 0
769                ELSE 1
770           END AS received_count,
771           CASE
772             WHEN (ro.memo IS NULL OR ro.memo = X'F6')
773               THEN 0
774             ELSE 1
775           END AS memo_present,
776           -- The wallet cannot receive transparent outputs in shielding transactions.
777           CASE
778             WHEN ro.pool = 0
779               THEN 1
780             ELSE 0
781           END AS does_not_match_shielding
782    FROM v_received_outputs ro
783    JOIN transactions
784         ON transactions.id_tx = ro.transaction_id
785    UNION
786    -- Outputs spent in this transaction
787    SELECT ro.account_id              AS account_id,
788           transactions.mined_height  AS mined_height,
789           transactions.txid          AS txid,
790           ro.pool                    AS pool,
791           id_within_pool_table,
792           -ro.value                  AS value,
793           0                          AS received_value,
794           ro.value                   AS spent_value,
795           1                          AS spent_note_count,
796           0                          AS change_note_count,
797           0                          AS received_count,
798           0                          AS memo_present,
799           -- The wallet cannot spend shielded outputs in shielding transactions.
800           CASE
801             WHEN ro.pool != 0
802               THEN 1
803             ELSE 0
804           END AS does_not_match_shielding
805    FROM v_received_outputs ro
806    JOIN v_received_output_spends ros
807         ON ros.pool = ro.pool
808         AND ros.received_output_id = ro.id_within_pool_table
809    JOIN transactions
810         ON transactions.id_tx = ros.transaction_id
811),
812-- Obtain a count of the notes that the wallet created in each transaction,
813-- not counting change notes.
814sent_note_counts AS (
815    SELECT sent_notes.from_account_id     AS account_id,
816           transactions.txid              AS txid,
817           COUNT(DISTINCT sent_notes.id)  AS sent_notes,
818           SUM(
819             CASE
820               WHEN (sent_notes.memo IS NULL OR sent_notes.memo = X'F6' OR ro.transaction_id IS NOT NULL)
821                 THEN 0
822               ELSE 1
823             END
824           ) AS memo_count
825    FROM sent_notes
826    JOIN transactions
827         ON transactions.id_tx = sent_notes.tx
828    LEFT JOIN v_received_outputs ro
829         ON sent_notes.id = ro.sent_note_id
830    WHERE COALESCE(ro.is_change, 0) = 0
831    GROUP BY account_id, txid
832),
833blocks_max_height AS (
834    SELECT MAX(blocks.height) AS max_height FROM blocks
835)
836SELECT accounts.uuid                AS account_uuid,
837       notes.mined_height           AS mined_height,
838       notes.txid                   AS txid,
839       transactions.tx_index        AS tx_index,
840       transactions.expiry_height   AS expiry_height,
841       transactions.raw             AS raw,
842       SUM(notes.value)             AS account_balance_delta,
843       SUM(notes.spent_value)       AS total_spent,
844       SUM(notes.received_value)    AS total_received,
845       transactions.fee             AS fee_paid,
846       SUM(notes.change_note_count) > 0  AS has_change,
847       MAX(COALESCE(sent_note_counts.sent_notes, 0))  AS sent_note_count,
848       SUM(notes.received_count)         AS received_note_count,
849       SUM(notes.memo_present) + MAX(COALESCE(sent_note_counts.memo_count, 0)) AS memo_count,
850       blocks.time                       AS block_time,
851       (
852            blocks.height IS NULL
853            AND transactions.expiry_height BETWEEN 1 AND blocks_max_height.max_height
854       ) AS expired_unmined,
855       SUM(notes.spent_note_count) AS spent_note_count,
856       (
857            -- All of the wallet-spent and wallet-received notes are consistent with a
858            -- shielding transaction.
859            SUM(notes.does_not_match_shielding) = 0
860            -- The transaction contains at least one wallet-spent output.
861            AND SUM(notes.spent_note_count) > 0
862            -- The transaction contains at least one wallet-received note.
863            AND (SUM(notes.received_count) + SUM(notes.change_note_count)) > 0
864            -- We do not know about any external outputs of the transaction.
865            AND MAX(COALESCE(sent_note_counts.sent_notes, 0)) = 0
866       ) AS is_shielding
867FROM notes
868LEFT JOIN accounts ON accounts.id = notes.account_id
869LEFT JOIN transactions
870     ON notes.txid = transactions.txid
871JOIN blocks_max_height
872LEFT JOIN blocks ON blocks.height = notes.mined_height
873LEFT JOIN sent_note_counts
874     ON sent_note_counts.account_id = notes.account_id
875     AND sent_note_counts.txid = notes.txid
876GROUP BY notes.account_id, notes.txid";

878/// Selects all outputs received by the wallet, plus any outputs sent from the wallet to
879/// external recipients.
880///
881/// This will contain:
882/// * Outputs received from external recipients
883/// * Outputs sent to external recipients
884/// * Outputs received as part of a wallet-internal operation, including
885///   both outputs received as a consequence of wallet-internal transfers
886///   and as change.
887///
888/// The `to_address` column will only contain an address when the recipient is
889/// external. In all other cases, the recipient account id indicates the account
890/// that controls the output.
891pub(super) const VIEW_TX_OUTPUTS: &str = "
892CREATE VIEW v_tx_outputs AS
893WITH unioned AS (
894    -- select all outputs received by the wallet
895    SELECT transactions.txid            AS txid,
896           ro.pool                      AS output_pool,
897           ro.output_index              AS output_index,
898           from_account.uuid            AS from_account_uuid,
899           to_account.uuid              AS to_account_uuid,
900           NULL                         AS to_address,
901           ro.value                     AS value,
902           ro.is_change                 AS is_change,
903           ro.memo                      AS memo
904    FROM v_received_outputs ro
905    JOIN transactions
906        ON transactions.id_tx = ro.transaction_id
907    -- join to the sent_notes table to obtain `from_account_id`
908    LEFT JOIN sent_notes ON sent_notes.id = ro.sent_note_id
909    -- join on the accounts table to obtain account UUIDs
910    LEFT JOIN accounts from_account ON from_account.id = sent_notes.from_account_id
911    LEFT JOIN accounts to_account ON to_account.id = ro.account_id
912    UNION ALL
913    -- select all outputs sent from the wallet to external recipients
914    SELECT transactions.txid            AS txid,
915           sent_notes.output_pool       AS output_pool,
916           sent_notes.output_index      AS output_index,
917           from_account.uuid            AS from_account_uuid,
918           NULL                         AS to_account_uuid,
919           sent_notes.to_address        AS to_address,
920           sent_notes.value             AS value,
921           0                            AS is_change,
922           sent_notes.memo              AS memo
923    FROM sent_notes
924    JOIN transactions
925        ON transactions.id_tx = sent_notes.tx
926    LEFT JOIN v_received_outputs ro ON ro.sent_note_id = sent_notes.id
927    -- join on the accounts table to obtain account UUIDs
928    LEFT JOIN accounts from_account ON from_account.id = sent_notes.from_account_id
929)
930-- merge duplicate rows while retaining maximum information
931SELECT
932    txid,
933    output_pool,
934    output_index,
935    max(from_account_uuid) AS from_account_uuid,
936    max(to_account_uuid) AS to_account_uuid,
937    max(to_address) AS to_address,
938    max(value) AS value,
939    max(is_change) AS is_change,
940    max(memo) AS memo
941FROM unioned
942GROUP BY txid, output_pool, output_index";

945    format!(
946        "CREATE VIEW v_sapling_shard_scan_ranges AS
947        SELECT
948            shard.shard_index,
949            shard.shard_index << 16 AS start_position,
950            (shard.shard_index + 1) << 16 AS end_position_exclusive,
951            IFNULL(prev_shard.subtree_end_height, {}) AS subtree_start_height,

964                scan_queue.block_range_start <= shard.subtree_end_height OR
965                shard.subtree_end_height IS NULL
